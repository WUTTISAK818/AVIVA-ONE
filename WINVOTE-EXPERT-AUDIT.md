WinVote มี **แนวคิดและการออกแบบที่ดีเยี่ยม** — การแยกมิติ "ตัวตน vs เจตนา", double opt-in กันโกง, โมเดลคะแนนเชิงความน่าจะเป็น, กระทบยอดสองฝั่ง, lifecycle — เป็นความคิดระดับมืออาชีพที่หายากในเครื่องมือลักษณะนี้ และโครงสร้างโค้ด/ฐานข้อมูลส่วนแกนสะอาด แข็งแรง

**แต่ปัญหาคือ "ช่องว่างระหว่างดีไซน์กับของจริง"** — กลไกความน่าเชื่อถือสำคัญส่วนใหญ่ยัง "อยู่บนกระดาษ ยังไม่ได้ build" ทำให้ ณ ขณะนี้:
- ลูกทีมทุจริต **สร้างฐานเสียงปลอม 50 คนได้** (ระบบรับเข้าเลย)
- ใครล็อกอินก็ **เห็นข้อมูลประชาชนทุกเขต**
- ตัวเลข "โอกาสชนะ" **ยังไม่ใช่ความน่าจะเป็นจริงเชิงสถิติ**

→ **เหมาะเป็นต้นแบบสาธิต (prototype) ยังไม่ควรใช้กับข้อมูลประชาชนจริงจนกว่าจะปิด 3 ช่องโหว่วิกฤต**

## 🔴 ช่องโหว่วิกฤต (Critical) — ผู้เชี่ยวชาญหลายท่านชี้ตรงกัน

### C1. ยังไม่มีการเทียบบัญชีผู้มีสิทธิ์ในโค้ดจริง → กันโกงไม่ทำงาน
*(Field + Data + Statistician)* — `validateThaiId` เช็คแค่ checksum ซึ่งสร้างเลขปลอมที่ผ่านได้ง่าย ไม่มีตาราง `voter_roll` ไม่มีการ match ในหน้าเก็บข้อมูล → **ลูกทีมกรอกเลขบัตรปลอม 50 ใบที่ระบบรับเข้าได้ทั้งหมด** — กลไกป้องกันการทุจริตหลักของทั้งระบบยังไม่มีผลบังคับ

### C2. RLS `using(true)` → ข้อมูลประชาชนทุกเขตเปิดให้ทุก login
*(Security + Field)* — `residents` เปิดให้ผู้ล็อกอินทุกคนอ่าน/แก้/ลบ เลขบัตร/รูปเซลฟี่/GPS ของทุกเขต RBAC คุมแค่ฝั่ง UI (เปิด DevTools/เรียก API ตรงก็ทะลุ) การแยกเขตมีแค่ comment ใน SECURITY-NOTES ยังไม่ apply

### C3. โมเดล "โอกาสชนะ" ยังไม่ใช่ความน่าจะเป็นจริง + ระบุคู่แข่งผิด
*(Statistician)* — ตัวเลข win% เป็นการ interpolate เชิงเส้นด้วยเลขกำหนดเอง (90/10/80) ไม่มี distribution/variance จริง · และเปรียบเทียบกับ **v1 อย่างเดียว** ทั้งที่ข้อมูลจริงหลายหน่วย **v4 ชนะ v1** (เช่น เขต2 หน่วย34: v1=38 แต่ v4=141) → ในระบบ plurality คู่แข่งที่แท้จริงคือ `max(v1,v3,v4)` การใช้ v1 ทำให้ **ประเมินโอกาสชนะสูงเกินจริง**

### C4. PDPA: ไม่มี consent/retention + เลขบัตร plaintext + บัญชีบนเครื่องลูกทีมไม่เข้ารหัส
*(Security)* — เก็บเลขบัตร/รูป/GPS โดยไม่มีบันทึกความยินยอม ไม่มีนโยบายลบ · เลขบัตรเก็บแบบ plaintext · แผนโหลดบัญชีผู้มีสิทธิ์ (เลข 13 หลัก) ลงมือถือลูกทีมยังไม่เข้ารหัส/ไม่ลบ → เครื่องหาย= ข้อมูลประชาชนรั่วจำนวนมาก · ส่งรูปบัตรไป OpenAI (ข้ามประเทศ) ไม่มีฐานทางกฎหมายระบุ

## 🟡 ควรแก้ (Medium)

- **คาลิเบรต/back-test สัญญาไว้แต่ยังไม่ทำ** — % ชั้น A/B/C เป็นค่า hardcode ไม่ได้ fit กับข้อมูล 185 หน่วยจริง *(Stat)*
- **เอกสาร repo ขัดกัน** — `AGENTS.md`/`CLAUDE.md` พูดถึงคนละโปรเจกต์ (`aviva-app-gpt`, dashboard v2.9.x) ไฟล์ pre-flight ที่บังคับให้อ่านไม่มีจริงในนี้ *(Code)*
- **รหัส `Demo1234` บน 5 บัญชีจริง** (รวม admin) — กันด้วย checklist มือ ไม่ได้บังคับในโค้ด *(Security)*
- **Rate limit ใช้ in-memory Map** — ไร้ผลบน serverless (Vercel หลาย instance) *(Security + Code)*
- **ตาราง reconciliation/lifecycle/voter_roll ยังไม่สร้าง** — กระทบยอด/attrition เป็น prose *(Data + Field)*
- **dedup ขัดกันเอง** — `national_id unique` (ปฏิเสธตัวซ้ำ) vs แผน "เก็บตัวซ้ำไว้ flag duplicate_contested" → ต้องใช้ partial unique `where claim_status='owned'` *(Data)*
- **trust_score ควร compute ไม่ใช่ store** (กัน stale) *(Data)*
- **chip reader ไม่ได้ตรวจ SW=9000 / ไม่จัดการ time-extension** + ยังไม่เทสต์ HW จริง *(Code)*

## 🟢 จุดแข็ง (เก็บไว้ ต่อยอด)

1. **โครงสร้างโค้ดสะอาด + TypeScript strict ผ่าน** — lib/UI/API แยกชั้นชัด, pattern ป้องกัน null สม่ำเสมอ *(Code 78)*
2. **schema normalize ดี** — keys/FK/constraint ถูกที่ (one-president partial index, on-delete ถูกต้อง) *(Data 71)*
3. **chip reader เป็น CCID/APDU จริงตามสเปกบัตรไทย** — ไม่ใช่ stub *(Code)*
4. **แนวคิด "ตัวตน ≠ เจตนา" + double opt-in** — ไอเดียที่ดีที่สุด จับ fraud vector ตรงจุด *(Field)*
5. **กรอบคะแนนเชิงความน่าจะเป็น** (คาดหวัง = turnout × P) — รากฐานถูกต้อง เหนือ "นับหัว = คะแนน" *(Stat)*
6. **มีข้อมูลจริง 185 หน่วยไว้คาลิเบรต** — สินทรัพย์ที่เครื่องมือทั่วไปไม่มี *(Stat/Data)*
7. **anon ถูกบล็อกสนิท + webhook ตรวจ signature + OCR ไม่เก็บรูปบัตร** *(Security)*
8. **แยกร่างจาก AVIVA สะอาดในระดับโค้ด/ข้อมูล** *(Code)*

## 🛣️ Roadmap จัดลำดับความสำคัญ

### P0 — ต้องทำก่อนแตะข้อมูลประชาชนจริง (ปิดช่องวิกฤต)
1. **สร้าง `voter_roll` + เทียบบัญชีในหน้าเก็บข้อมูล** — gate การบันทึกด้วยสถานะ in_unit/other_unit/not_found + ดึงชื่อทางการจากบัญชี *(แก้ C1)*
2. **เปลี่ยน RLS เป็นแยกตามเขตจริง** — สร้าง `user_districts`, apply policy, ทดสอบด้วย JWT non-admin ยิง API ตรง *(แก้ C2)*
3. **ทำ DEMO_MODE ให้เป็นไปไม่ได้บน production** (fail build ถ้าตั้งคู่กับ Supabase จริง) *(แก้ H3)*

### P1 — ก่อน go-live
4. **เปลี่ยนโมเดลโอกาสชนะเป็น Monte Carlo + Beta/Binomial** เทียบ `max(v1,v3,v4)` ต่อหน่วย + คาลิเบรตจาก 185 หน่วย (Brier/log-loss) *(แก้ C3 + Stat P0)*
5. **PDPA: ตาราง consent + นโยบายลบ + เข้ารหัสบัญชีบนเครื่อง + DPA สำหรับ OpenAI** *(แก้ C4)*
6. **ทำ `intent_status` + double opt-in จริง** (เจ้าตัวกด ✅/❌) แยกจาก phone_verified *(แก้ H2)*
7. **แก้ LINE expires_at** + ปิด public signup + ลบ/เปลี่ยนรหัส demo (บังคับในโค้ด/CI) *(แก้ H1 + Medium)*

### P2 — คุณภาพ/ความครบ
8. **Offline queue (IndexedDB) + per-record receipt** (ได้ทั้ง offline + ฐานของ reconciliation funnel) — แก้ปม KillServiceWorker ก่อน *(แก้ H4)*
9. **โหมดง่ายลูกทีมสูงอายุ** (ถ่ายบัตร→เบอร์→ส่ง, defer OCR ตอน sync) *(แก้ H5)*
10. **ตาราง lifecycle + audit log + reconciliation_snapshot + dashboard หัวหน้าทีม**
11. แก้เอกสาร repo (AGENTS/CLAUDE ให้ตรง WinVote), rate limit ใช้ store จริง, chip reader ตรวจ SW=9000, ทำความสะอาด dead code

## สรุปจากหัวหน้าคณะ

WinVote คือ **โครงการที่ "คิดมาดี" อย่างหายาก** — ดีไซน์เข้าใจปัญหาเลือกตั้งจริง ทั้งการกันโกง ความเป็นส่วนตัว และสถิติ ถ้าวัดที่ "พิมพ์เขียว" จะได้คะแนนสูงกว่านี้มาก

แต่หน้าที่ของคณะคือประเมิน **"ของจริงวันนี้"** — ซึ่งยังเป็นต้นแบบที่กลไกสำคัญหลายอย่างยังไม่ทำงาน ช่องว่างที่อันตรายที่สุดคือ **โค้ดยังไม่บังคับสิ่งที่ดีไซน์สัญญาไว้** ทำให้วันนี้ยังกันการสร้างฐานเสียงปลอมไม่ได้ และข้อมูลประชาชนยังเปิดกว้างเกินไป

**คำแนะนำ:** ปิด **P0 ทั้ง 3 ข้อ** ให้ได้ก่อนนำไปใช้กับข้อมูลจริง แล้วระบบนี้มีศักยภาพขึ้นถึงระดับ "ดีมาก" ได้จริงตามที่ดีไซน์วางไว้
