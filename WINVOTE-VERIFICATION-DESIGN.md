# WinVote — ดีไซน์ระบบยืนยันฐานเสียง (Verification Design)

> **สถานะ:** เอกสารดีไซน์ (พิมพ์เขียว) — ยังไม่ลงมือ build
> บันทึกผลการออกแบบร่วมกับผู้ใช้ ก่อนลงมือทำจริงตอน Supabase project WinVote กลับมา ACTIVE
> อัปเดตล่าสุด: ระหว่าง session ออกแบบ (ก่อน DB ตื่น)

เอกสารนี้เป็น **บันทึกการตัดสินใจ** เรื่องการเก็บ + ยืนยันข้อมูลฐานเสียง เพื่อกันข้อมูลหายและใช้เป็นพิมพ์เขียวตอน implement จริง

---

## 1. โมเดลทีม

```
หัวหน้าทีม (members.member_role = 'president')
└─ ลูกทีม (members.member_role = 'team') — รับผิดชอบฐานเสียง 50 คน (resident_quota = 50)
      └─ ผู้มีสิทธิ์ 50 คน (residents)
```
✅ schema เดิมรองรับแล้ว (`member_role`, `resident_quota`)

**เป้าหมายของลูกทีม:** พิสูจน์ว่าฐานเสียง 50 คน (ก) มีตัวตนจริง (ข) เป็นผู้มีสิทธิ์เลือกตั้งครั้งนี้ (ค) สนับสนุนเราจริง

---

## 2. แนวคิดหลัก: ยืนยัน 2 มิติ (แยกจากกัน)

| มิติ | ถามว่า | พิสูจน์ด้วย |
|---|---|---|
| **① ตัวตน (Identity)** | คนจริง + มีสิทธิ์จริงไหม | เลข 13 หลัก + เทียบบัญชี + ชิป/รูป |
| **② เจตนา (Intent/ยินยอม)** | เจ้าตัวรับรู้ + สนับสนุนจริงไหม | เบอร์โทร → เจ้าตัวยืนยันเอง |

> บัตรถูก + อยู่ในบัญชี = ตัวตนจริง **แต่ไม่ได้แปลว่าเจ้าตัวสนับสนุน**
> (เคสจริง: พ่อแม่เอาบัตรลูกมาให้ ลูกไม่รู้เรื่อง) → ต้องมีมิติเจตนาแยกต่างหาก

---

## 3. วิธีเก็บข้อมูล (3 โหมด ตามอุปกรณ์/ความถนัด)

| โหมด | อุปกรณ์ | ความแม่นชื่อ | ออฟไลน์ |
|---|---|---|---|
| **เสียบชิป** | Android + เครื่องอ่าน CCID | 100% (อ่านจากชิป) | ✅ ไม่ต้องใช้เน็ต |
| **โหมดง่าย: ถ่ายบัตร + กรอกเบอร์ + กดส่ง** | มือถือใดก็ได้ (เหมาะลูกทีมสูงอายุ) | ดี (ดูข้อ 6) | ✅ เก็บรูปไว้ก่อน |
| **พิมพ์เลขบัตรเอง** | สำรองสุด | — | ✅ |

**ข้อจำกัดที่ต้องรู้:**
- เสียบชิป: ใช้ได้เฉพาะ Chrome/Edge บน **Android/Desktop** — **iPhone/Safari ไม่รองรับ WebUSB**
- ถ่ายรูป OCR (OpenAI Vision): **ต้องมีเน็ต** → ตอนออฟไลน์ให้เก็บรูปไว้ ประมวลผล OCR ตอน sync
- โค้ดอ่านชิป (`thai-id-reader.ts`) + OCR (`api/winvote/extract-id`) เขียนครบแล้ว แต่ชิป **ยังไม่ได้เทสต์กับฮาร์ดแวร์จริง**

---

## 4. เทียบบัญชีผู้มีสิทธิ์ (Voter Roll Cross-check)

- บัญชีรายชื่อ = ราชการส่งมาทุกครั้งที่เลือกตั้ง **มีเลข 13 หลัก + แยกตามเขต/หน่วย** (บอกว่าแต่ละคนอยู่หน่วยไหน เขตอะไร)
- โหลดบัญชีลงเครื่อง → เทียบเลขบัตร **ทำออฟไลน์ได้** (ไม่ต้องเน็ต ไม่ต้องเครื่องอ่าน ได้ทุกมือถือ)

```
เก็บเลขบัตร → เทียบบัญชี →
   ✅ อยู่ในบัญชี หน่วยนี้/เขตนี้   → ฐานเสียงถูกต้อง (main list)
   ⚠️ อยู่ในบัญชี แต่คนละหน่วย     → เก็บเข้า "บัญชีพิเศษ" (special list)
   ❌ ไม่อยู่ในบัญชีเลย            → เตือน "ไม่มีสิทธิ์ครั้งนี้" ไม่เก็บ
```

> ⚠️ **อย่าตัดสินนอกเขตจากที่อยู่บนบัตร** (ทะเบียนบ้านอาจไม่ตรงสิทธิเลือกตั้ง) — ยึด **บัญชีรายชื่อ** เป็นความจริงเสมอ

---

## 5. ความถูกต้องของชื่อ — แก้ด้วยบัญชี (สำคัญ)

สำหรับคนที่ **อยู่ในบัญชี**: ไม่ต้องพึ่ง OCR ให้อ่านชื่อเป๊ะ — แค่ต้องได้ **เลข 13 หลักที่ถูก** (checksum ตรวจด้วย `validateThaiId`) แล้ว:

```
ถ่ายบัตร → OCR อ่านเลข 13 หลัก → checksum ผ่าน → เทียบบัญชี → เจอ
   → ดึง "ชื่อทางการจากบัญชี" มาใช้   ← สะกดถูก 100% เสมอ ไม่ว่ารูปเบลอแค่ไหน
```

→ **ชื่อในระบบสะกดถูก 100% ทุกคนที่อยู่ในบัญชี** (เสียบชิปก็ได้ชื่อเป๊ะเหมือนกัน แต่โหมดถ่ายรูปไม่ต้องมีเครื่องอ่าน)

---

## 6. มิติเจตนา — ยืนยันว่าสนับสนุนจริง (กันเอาบัตรคนอื่นมามั่ว)

### ⭐ Double Opt-in (ยืนยันสองทาง)
ระบบส่งข้อความไปเบอร์/LINE ของ **เจ้าตัวเอง**:
> *"คุณได้ลงทะเบียนสนับสนุน [ผู้สมัคร] ผ่าน [ลูกทีม X] — กด ✅ ยืนยัน / ❌ ไม่ใช่ฉัน"*
- เจ้าตัวกดยืนยัน = สนับสนุนจริง รับรู้จริง (ต่อยอดจากโค้ด LINE verify ที่มีอยู่)
- กดปฏิเสธ/เงียบ → ขึ้นธง "ยังไม่ยืนยันเจตนา" + เตือนหัวหน้าทีม
- ลูกทีมเอาบัตรคนอื่นมามั่ว → เจ้าตัวได้ข้อความแล้วกดปฏิเสธ → **จับโกหกได้ทันที**

### เสริม
- สถานะเจตนา 3 ระดับ: `confirmed / pending / rejected`
- สุ่มโทรตรวจ (audit): สตาฟสุ่ม 5/50 โทรยืนยันซ้ำ
- ไม่มีเบอร์ก็เก็บได้ แต่ trust score ต่ำ + ขึ้นธง "ต้องลงพื้นที่ยืนยัน"
- **เบอร์โทร = ขอให้กรอกทุกคน** (ใช้ยืนยันเจตนา + รีเช็คภายหลัง)
- Consent (PDPA) + ยืนยันเจตนา ทำในจังหวะกดเดียวกันได้

---

## 7. กันข้อมูลซ้ำ (ลูกทีมเก็บคนเดียวกัน)

- **dedup ด้วยเลข 13 หลัก** (national_id unique มีแล้ว) — ไม่ใช่ชื่อ (ชื่อเพี้ยน/ซ้ำได้)
- **เจ้าของ = `captured_at` เร็วสุด** (คนที่เจอตัวจริงก่อน) — ไม่ใช่เวลา sync (ออฟไลน์ทำให้คนเก็บก่อนอาจ sync ทีหลัง)
  - ระวังนาฬิกาเครื่องเพี้ยน → เซิร์ฟเวอร์เก็บเวลา sync คู่กัน ถ้า captured_at ผิดปกติให้ขึ้นธง
- **แก้ไข/เปลี่ยนเจ้าของ = เฉพาะสตาฟระดับสูง** (rbac: `canApprove`/`canEditData`)
- ออฟไลน์: ตรวจซ้ำไม่ได้ทันที → เจอตอน sync → First-captured-wins + แจ้งเตือนทั้งสองฝั่ง

---

## 8. กระทบยอดสองฝั่ง (Reconciliation) — ปัญหา "ลูกทีมบอก 50 ระบบมี 47"

### Verification Funnel (ต่อลูกทีม) — ทำให้ "ช่องทางหาย" มองเห็นได้
```
เป้า 50
├─ บันทึกในเครื่อง (captured)   50
├─ ส่งเข้าระบบ (synced)         48   ← 2 ค้างคิวออฟไลน์ (เน็ต)
├─ ผ่านเกณฑ์ (accepted)         46   ← 2 ตัดออก (นอกบัญชี/ซ้ำ)
├─ ยืนยันเจตนา (confirmed)      41   ← 5 รอเจ้าตัวยืนยัน
└─ ✅ ใช้งานได้ (effective)     41
```

### 3 กลไกให้ยอดตรงเสมอ
1. **ใบยืนยันรายรายการ** — sync แล้วเซิร์ฟเวอร์ตอบว่ารับ/ไม่รับรายไหนเพราะอะไร → เครื่องกระทบยอดทันที ไม่มีอะไรหายเงียบ
2. **คิวออฟไลน์ไม่ทิ้งของ** — รายส่งไม่ผ่านค้างในเครื่องพร้อมเหตุผล กดส่งซ้ำได้
3. **ปิดยอดรายงวด + เซ็นรับสองฝั่ง** — ระบบ freeze ยอด → ลูกทีมกดรับรอง/โต้แย้ง → หัวหน้าอนุมัติ → ถ้าโต้แย้งเปิดเคส dispute ตรวจ sync log (เหมือนกระทบยอด statement ธนาคาร)

---

## 9. วงจรชีวิตฐานเสียง (Lifecycle/Attrition) — ปัญหา "ส่ง 50 แต่ภายหลังเหลือ 44"

### สถานะ resident (เปลี่ยนได้ + มีประวัติ)
```
active        ใช้งานได้
disqualified  ขาดคุณสมบัติ (เสียชีวิต/ย้าย/ถูกเพิกถอนสิทธิ/หลุดบัญชีรอบใหม่)
withdrawn     ถอนตัว/ปฏิเสธสนับสนุนภายหลัง
pending       รอยืนยัน
```

### หลักการ
1. **ห้ามลบจริง** — soft status + audit log (ใคร-เมื่อไร-เพราะอะไร)
2. **แยก Gross (เก็บมา 50) กับ Effective (ใช้งานได้จริง 44)** — แสดงคู่กัน ส่วนกลางใช้ effective วางแผน
3. **เหตุผลมาตรฐาน (dropdown)** — วิเคราะห์สถิติได้
4. **ปลด/ตัดออก = สตาฟระดับสูง หรือระบบอัตโนมัติ** (เช่น import บัญชีรอบใหม่แล้วชื่อหาย) — **ไม่ให้ลูกทีมปลดเอง** (กันผลประโยชน์ทับซ้อน)
5. **เติมทดแทน (Replenishment)** — effective ต่ำกว่า 50 → ลูกทีมเติมใหม่ได้
6. **รอบยืนยันซ้ำ (Re-confirmation)** — ใกล้เลือกตั้งให้ระบบ trigger ยืนยันเจตนาซ้ำ (freshness) → effective แม่นถึงวันจริง

---

## 10. Trust Score (รวม 2 มิติ)

ตัวอย่างการให้น้ำหนัก:
```
ตัวตน:  เสียบชิป/อยู่ในบัญชี (50) + เซลฟี่+GPS (20)
เจตนา:  เจ้าตัวยืนยัน (30)
รวม = 100%
```
→ โปร่งใส วัดได้ หัวหน้าทีมเห็นว่าใคร "ครบ" ใคร "อ่อน"

## 11. ออฟไลน์ (Offline-first)

- เก็บข้อมูลได้แม้ไม่มีเน็ต → เก็บลงเครื่อง (queue) → sync เมื่อมีเน็ต (กดเอง **หรือ** อัตโนมัติเมื่อออนไลน์)
- เทียบบัญชี + กันซ้ำในเครื่อง (เท่าที่มีข้อมูล local) → ตรวจซ้ำข้ามลูกทีมทำตอน sync
- OCR/ยืนยันเจตนา ทำตอน sync (ต้องใช้เน็ต)
- ⚠️ หมายเหตุเทคนิค: ปัจจุบันมี `KillServiceWorker.tsx` (ฆ่า SW) — ถ้าจะทำ offline จริงต้องมี service worker/cache ตรงข้ามกับของเดิม ต้องทบทวน
- ⚠️ PDPA: บัญชี (เลข 13 หลัก) ที่โหลดลงเครื่องลูกทีม = PII — ควรโหลดเฉพาะเขต/หน่วยที่ลูกทีมรับผิดชอบ, เข้ารหัสที่เก็บ, ลบหลังเลือกตั้ง

## 12. สรุปโครงข้อมูลที่ต้องเพิ่ม (รวมทุกข้อ)

```sql
-- ตารางใหม่
winvote.voter_roll (
  election, national_id, full_name,
  district_code, polling_unit_no,
  unique(election, national_id)
)

winvote.resident_status_log (        -- ประวัติเปลี่ยนสถานะ (ห้ามลบ)
  resident_id, from_status, to_status, reason, by_user, at
)

winvote.reconciliation_snapshot (    -- ปิดยอดรายงวด เซ็นสองฝั่ง
  member_id, period, captured, synced, accepted, confirmed,
  gross, effective, leader_signoff, central_approved, at
)

-- residents เพิ่ม field
  capture_method   : chip | photo | manual
  roll_status      : in_unit | other_unit | not_found
  list_type        : main | special
  claim_status     : owned | duplicate_contested
  intent_status    : confirmed | pending | rejected
  status           : active | disqualified | withdrawn | pending
  status_reason    : text
  last_confirmed_at: timestamptz
  trust_score      : int (0-100)
  synced           : boolean
```
> ยอดใน funnel คำนวณจาก field เหล่านี้ ไม่ต้องกรอกซ้ำ

## 13. การตัดสินใจที่ตกลงแล้ว (สรุป)

| # | ตกลงเป็น |
|---|---|
| คนละหน่วย | แยกเป็น **บัญชีพิเศษ** นับไว้ ไม่เสียแม้แต่เสียงเดียว |
| ซ้ำ | `captured_at` ก่อน = เจ้าของ \| แก้ไขเฉพาะสตาฟระดับสูง |
| นอกบัญชีจริง | เตือน ไม่เก็บ (เก็บแค่ตัวนับสถิติ) |
| เบอร์โทร | ขอให้กรอกทุกคน (ยืนยันเจตนา) มีก็ดี ไม่มีได้แต่ flag |
| Offline | กดส่งเอง / sync เอง / อัตโนมัติเมื่อออนไลน์ — ได้หมด |
| โหมดง่าย | ถ่ายบัตร+เบอร์+กดส่ง สำหรับลูกทีมสูงอายุ |

## 14. ขั้นต่อไป (ตอน DB ตื่น)
1. สร้าง schema เพิ่ม (voter_roll + fields + log + snapshot)
2. ทำ import บัญชีผู้มีสิทธิ์ (แยกเขต/หน่วย)
3. โหมดเก็บข้อมูล 3 แบบ + เทียบบัญชี + ดึงชื่อทางการ
4. กันซ้ำ + บัญชีพิเศษ
5. ยืนยันเจตนา (double opt-in ต่อยอด LINE)
6. Offline queue + sync + ใบยืนยันรายรายการ
7. หน้ากระทบยอด (funnel) + ปิดยอดเซ็นสองฝั่ง
8. Lifecycle + audit log + re-confirmation

> ยังมีหัวข้อที่คุยต่อได้: รายละเอียด UX แต่ละโหมด, สูตร trust score จริง, กลไก audit/สุ่มตรวจ, PDPA consent flow
